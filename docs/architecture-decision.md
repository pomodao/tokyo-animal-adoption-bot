# アーキテクチャ方針決定

## 目的

東京都動物愛護相談センターの譲渡動物ページを定期的に確認し、更新があれば SNS に投稿する bot を、できるだけ低コストで構築する。

対象ページ:

- https://shuyojoho.metro.tokyo.lg.jp/generals/
- https://shuyojoho.metro.tokyo.lg.jp/generals/cat

## 要件

- 固定費が低い、またはほぼゼロであること
- 定期実行できること
- 前回スナップショットや重複投稿防止用の状態を保持できること
- SNS 投稿との接続が単純であること
- 運用負荷が低いこと
- 可能なら掲載画像も取得して投稿できること

## 検討した選択肢

### 1. GitHub Actions + リポジトリ内状態管理

利点:

- 公開リポジトリなら無料で使える場合がある
- 始めやすい
- 小さなスクリプトの定期実行には向いている
- 状態ファイルを git にコミットすれば変更履歴をそのまま監査できる
- 外部の永続ストアがなくても最小構成を組める

懸念:

- 定期実行にはプラットフォーム固有の制約がある
- 公開リポジトリでは非アクティブ期間が続くと scheduled workflow が無効化されうる
- `repository activity` の厳密条件は GitHub Docs で明示されていない
- 60 日ルールは commit 以外の活動では確実に回避できるとは言い切れない
- 状態コミットが増えるため、通常のソース変更履歴と運用履歴が同居する

### 2. VPS または常時稼働コンテナ

利点:

- 実行環境を完全に制御できる
- ローカル状態の保持が単純

懸念:

- 通常は毎月の固定費が発生する
- この用途に対して運用負荷が高い

### 3. Cloudflare Workers + Cron Triggers + KV

利点:

- 軽量な定期ポーリング処理と相性が良い
- 常時稼働サーバーが不要
- スケジューラを標準で持つ
- スナップショットや投稿済み情報を KV に簡潔に保持できる
- 運用負荷が低い

懸念:

- ランタイム制約の範囲で単純な実装に寄せる必要がある
- フルサーバー環境より HTML パースの選択肢が少し狭い

## 採用方針

次の構成を採用する。

- 実行基盤: `GitHub Actions`
- 定期実行: `schedule` event
- 状態保存: リポジトリ内の管理用 JSON ファイル
- 主投稿先: `Bluesky`
- 任意の追加投稿先: `X`
- 60 日対策: 事前警告 Issue の作成、および必要時の手動再有効化

## 採用理由

この構成は、「できるだけコストをかけない」という最重要条件を最も強く満たす。

想定される処理は小さい。

- 1 ページ取得する
- 一覧をパースする
- 前回状態と比較する
- 新規掲載があったときだけ投稿する

この規模であれば、GitHub Actions で十分に処理できる。加えて、状態ファイルをリポジトリで管理すれば、別途 KV やデータベースを持たずに運用を開始できる。

掲載画像の取得が必要になっても、一覧または詳細ページから画像 URL を取得し、実行時にダウンロードして SNS へ添付する構成であれば、引き続き GitHub Actions で対応できる。

Cloudflare Workers 構成は技術的にはきれいだが、今回の優先順位では、外部ストア込みの別基盤を追加しない GitHub Actions 構成の方がより低コストである。

## この判断によって決まること

- 差分判定は HTML 全体ではなく、正規化済みレコード同士で行う
- 安定した識別子として `管理番号` を使う
- 最初に対応するイベントは `新規掲載` のみとする
- 削除通知や既存レコードの項目変更通知は初期スコープに含めない
- SNS 投稿処理は差分判定処理から分離し、後から投稿先を増やしやすくする
- 監視状態はリポジトリ内ファイルに保存し、更新時にはコミットする
- state ファイルは毎回書き換えず、スナップショット差分や投稿結果の増加など意味のある変化があるときだけ更新する
- 画像情報は別ファイルへ分離せず、動物スナップショットへ集約する
- 可能なら詳細ページまたは一覧から画像 URL を取得し、投稿時に添付する
- public repository で 60 日無活動が続く可能性を前提に、警告と手動対応を運用へ組み込む
- 連続失敗の通知は GitHub Actions の run 履歴を使って判定し、Issue で管理する
- 投稿済み状態は重複防止のため永続保持し、当面は自動 pruning を行わない

## 見直し条件

次のいずれかが発生したらこの方針を見直す。

- 対象サイトが JavaScript レンダリング必須になった
- 更新頻度や処理量が大きく増えた
- `X` が必須の主投稿先になった
- 状態コミットがリポジトリ運用のノイズとして無視できなくなった
- 60 日ルールへの手動対応が許容しにくくなった
- 画像取得コストや添付制約により Actions 実行時間や実装複雑性が大きく増えた
