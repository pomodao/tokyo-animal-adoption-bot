# アーキテクチャ

## 概要

bot は `GitHub Actions` の定期実行で起動し、東京都動物愛護相談センターの譲渡猫一覧ページを取得して、動物情報を正規化したレコードへ変換する。前回スナップショットと比較し、新しく掲載された動物が見つかった場合だけ SNS に投稿する。最新状態と投稿済み情報はリポジトリ内の管理用ファイルとして保持し、意味のある状態変化があったときだけ git へコミットする。失敗通知は state ファイルではなく GitHub Actions の run 履歴と Issue で扱う。

## 構成要素

### 1. スケジューラ

`GitHub Actions` の `schedule` event が一定間隔でジョブを起動する。

初期設定として推奨する監視間隔:

- `2〜3時間` ごと

補足:

- scheduled workflow は default branch 上で動かす
- public repository では 60 日間 repository activity がないと workflow が無効化されうる

### 2. 実行基盤

`GitHub-hosted runner` 上で TypeScript の監視スクリプトを直接実行する。

責務:

- 対象ページの取得
- 動物一覧のパース
- 必要に応じた詳細ページの取得
- レコードの正規化
- 掲載画像 URL の抽出
- 投稿用画像の取得
- 前回状態の読み込み
- 差分計算
- SNS 投稿
- 新しい状態の保存

実行方針:

- `tsc --noEmit` で型チェックする
- `node` で `.ts` エントリポイントを直接実行する
- 生成済み `dist/` は前提にしない

### 3. 状態保存

リポジトリ内の管理用ファイルに、掲載内容と投稿済み状態に関する最小限の状態を保存する。

想定ファイル:

- `state/cat.json` - 最新の正規化スナップショット
- `state/posted.json` - 投稿済み管理番号の一覧

### 4. 投稿レイヤー

投稿先ごとの実装は共通インターフェースの背後に分離する。

初期対応:

- `BlueskyPublisher`

任意対応:

- `XPublisher`

### 5. 運用監視レイヤー

60 日無活動による scheduled workflow 無効化リスクを下げるため、監視本体とは別に運用補助ジョブを置く。

役割:

- default branch の最終コミット日時を確認する
- しきい値を超えたら管理者向け Issue を作成する
- 必要なら手動対応を促す
- monitor workflow の連続失敗を GitHub Actions の run 履歴から判定する
- しきい値を超えたら障害 Issue を作成または更新する
- 復旧したら障害 Issue をクローズする

初期しきい値:

- `45日` 経過で警告
- `3回` 連続失敗で障害通知

## データフロー

1. scheduled workflow が監視ジョブを起動する
2. ジョブが `https://shuyojoho.metro.tokyo.lg.jp/generals/cat` を取得する
3. パーサーが次のような項目を抽出する
   - 管理番号
   - 名前
   - 管理支所
   - 詳細 URL があればその URL
4. 必要なら詳細ページへアクセスし、画像 URL を抽出する
5. 抽出結果を決定的な形式へ正規化する
6. リポジトリ内の状態ファイルから前回スナップショットを読み込む
7. `管理番号` を基準に新規掲載レコードを検出する
8. 投稿済みでない新規レコードごとに以下を行う
   - 投稿文を生成する
   - 添付可能な画像があれば取得する
   - 有効な SNS 投稿先へ投稿する
   - 重複防止マーカーを書き込む
9. スナップショット差分、投稿結果の増加があった場合だけ該当 state ファイルを保存する
10. state ファイルに変更がある場合だけ default branch へコミットする
11. workflow 結果をもとに連続失敗数を判定し、必要なら Issue を作成または更新する

## 正規化レコードの例

```json
{
  "id": "25ネ16",
  "name": "りゅう",
  "branch": "本所",
  "detailUrl": "https://shuyojoho.metro.tokyo.lg.jp/generals/detail/...",
  "imageUrl": "https://shuyojoho.metro.tokyo.lg.jp/...",
  "sourceUrl": "https://shuyojoho.metro.tokyo.lg.jp/generals/cat",
  "firstSeenAt": "2026-04-19T06:00:00.000Z"
}
```

`firstSeenAt` は、この bot がその動物を最初に検知した時刻であり、画像取得日時や掲載日時そのものではない。

`posted.json` は重複投稿防止のための永続状態として扱う。現在の実装では古い投稿済み記録を自動削除しないため、データは時間とともに増加する。容量や運用ノイズが無視できなくなった時点で pruning 方針を再検討する。

## 差分判定方針

初期段階では、差分判定の対象を意図的に絞る。

通知するもの:

- `管理番号` が新しく現れたレコード

初期段階では通知しないもの:

- 一覧からの削除
- 並び順の変更
- 見た目だけの HTML 変更
- 既知レコードの軽微な項目修正

## 投稿方針

投稿文は短く保ち、元ページへ戻れるリンクを必ず含める。

含める候補:

- 情報元の名称
- 新規掲載件数
- 管理番号
- 名前
- 管理支所
- 一覧ページ URL

画像投稿方針:

- 画像 URL を取得できる場合は画像付き投稿を優先する
- 画像取得に失敗した場合でも、本文のみで投稿できるようにする
- 投稿先ごとの添付制約に応じて、必要ならリサイズや添付枚数制限を考慮する

## GitHub Actions 運用方針

- 監視用 workflow と運用警告用 workflow は分離する
- state ファイルを更新したときだけコミットする
- `GITHUB_TOKEN` で state 更新コミットを行う場合、追加の workflow 連鎖は前提にしない
- workflow が 60 日ルールで無効化された場合は GitHub UI から `Enable workflow` して復旧する
- 更新がない通常実行では commit を発生させない
- 連続失敗通知は `monitor.yml` の run 履歴を参照して判定する
- 障害通知は Issue を開閉して管理し、失敗のたびに state commit は行わない

## 60 日無活動への対応

GitHub Docs は `repository activity` の厳密な定義を明示していない。そのため、運用上は `default branch の最新コミット日時` を代理指標として扱う。

初期運用ルール:

- 最終コミットから `45日` を超えたら警告 Issue を作成する
- 警告 Issue では、60 日到達前に管理者が状況を確認できるようにする
- 無効化されても GitHub UI から再有効化して運用を継続する
- Issue 作成や手動の Issue 操作は、60 日ルール回避策としては前提にしない

管理者向け Issue に含める内容の例:

- 最終コミット日時
- 現在の経過日数
- workflow が無効化される可能性があること
- 必要なら手動で再有効化する手順への案内

## 信頼性に関するルール

- 取得に失敗した場合は前回スナップショットを上書きしない
- パース結果が不自然な場合は投稿しない
- 投稿済みマーカーは対象 SNS への投稿成功後に保存する
- 1 つの SNS 投稿失敗が他の投稿先を必ず止める設計にはせず、必要なら投稿先ごとに失敗を切り分ける
- state 保存とコミットは、投稿処理の成否と矛盾しない順序で行う
- 画像取得に失敗しても、画像なし投稿へ安全にフォールバックできるようにする
- 画像 URL の変化だけで不要な再投稿が起きないようにする
- 取得結果に差分がない成功 run では state を書き換えない
- 失敗通知は run 履歴ベースで行い、失敗のたびに state ファイルを書き換えない

## セキュリティ

- SNS の認証情報は GitHub Actions Secrets に保存する
- トークン、アプリパスワード、セッション情報をリポジトリへ含めない
